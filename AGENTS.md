# AGENTS.md

This file provides guidance to Codex (Codex.ai/code) when working with code in this repository.

## Commands

```bash
npm run dev       # start Vite dev server on port 5173
npm run build     # production build → dist/
npm run preview   # serve the dist/ build locally
```

No test runner or linter is configured. There are no pre-commit hooks. Commits require bypassing GPG signing in this environment: `git -c commit.gpgsign=false commit`.

The built-in preview pane is wired via `.Codex/launch.json` — it starts `npm run dev` automatically.

## Architecture

**Stack:** Vite + React 18 (JSX), React Router v6, Supabase JS client. No TypeScript, no CSS modules — all styles are inline JS objects.

**Routing** (`src/App.jsx`):
- `/` → HomePage, `/builds` → BuildsPage, `/builds/:slug` → DetailsPage
- `/wiki` → WikiPage, `/wiki/:slug` → WikiArticlePage
- `/submit` → SubmitBuildPage, `/submit-wiki` → SubmitWikiPage

**Design system** — two sources of truth that must stay in sync:
- `src/tokens.js` — JS constants (`KW` color object, `TINT` map, `tagKind()`, `LOGO_COLORS`)
- `src/tokens.css` — CSS custom properties (`--kw-mono`, `--kw-lavender`, etc.) + Anonymous Pro `@font-face`

All components use `KW.*` from `tokens.js` for inline styles. `var(--kw-mono)` is used for the `font` shorthand string. Font files live in `public/` (served at `/`).

**Data layer** (`src/lib/supabase.js`):
- Single Supabase client (anon/publishable key — safe to expose, hardcoded directly in the file, no `.env` needed)
- Build helpers: `fetchBuilds({ q, filter })`, `fetchBuildBySlug(slug)`, `getArt(build)`, `getLayoutCode(layout)`, `getBuildTags(build)`, `buildSlug(name)`
- Wiki helpers: `fetchWikiArticles({ category, status, limit })`, `fetchWikiArticleBySlug(slug)`, `searchWikiArticles(q)`, `submitWikiArticle(data)`
- `CATEGORY_SLUG_MAP` is internal (not exported) — maps display labels (e.g. `'beginner guides'`) to DB slugs (e.g. `'beginner-guides'`); `submitWikiArticle` applies this mapping and always inserts with `status: 'pending'`
- `getArt(build)` falls back to deriving palette from `case_material` when `build.art` is null (polycarbonate→blue, brass→lavender, carbon→slate, steel→olive, aluminum→pink, else cream)
- `fetchBuildBySlug(slug)` uses `.or(slug.eq.X, name.ilike.X)` — name fuzzy match is a fallback for slugs that don't match exactly

**KeebArt** (`src/components/KeebArt.jsx`) — procedural keyboard illustration rendered as a CSS grid of colored divs. Props: `palette` (lavender/pink/blue/cream/olive/slate), `layout` (40/60/65/75/tkl), `seed` (integer for deterministic randomness). No real photos are used anywhere — all keyboard visuals are generated by this component.

**Tag coloring** — `tagKind(text)` in `tokens.js` classifies a tag string into `layout | switches | materials`, which maps to a `TINT` entry. The `Tag` component uses this for colored pills.

## Supabase database

Two tables. SQL files must be run manually in the Supabase SQL Editor (network restrictions prevent running migrations from this environment):

- `supabase/seed.sql` — `builds` table + RLS + 15 seed builds. The `slug` column is computed (`GENERATED ALWAYS AS`).
- `supabase/wiki.sql` — `wiki_articles` table + RLS + `updated_at` trigger + 9 seed articles.

**`builds` columns of note:** `art TEXT` (palette name for KeebArt), `sound_signature TEXT[]`, `typing_feel TEXT[]`, `photos TEXT[]` (Supabase Storage URLs), `switch_type TEXT` (enum: linear/tactile/clicky).

**`wiki_articles` columns of note:** `category TEXT` (enum: beginner-guides/modding-guides/parts-glossary/sound-feel/community-buying/about), `format TEXT` (sections/combined), `content JSONB` (array of `{heading, body, bullets[], tip}` for sections format), `combined_content TEXT` (for combined format), `status TEXT` (draft/pending/published). RLS allows anyone to read `published` and insert `draft/pending`.

**Dollar signs in wiki.sql seed data:** The Supabase SQL editor misinterprets `$50`, `$100` etc. inside dollar-quoted strings as query parameters. Escape them as the JSON unicode sequence `$` within `$j$...$j$` JSONB literals — PostgreSQL's JSONB parser decodes `$` to `$` correctly.

## Key conventions

- No TypeScript, no prop-types — components are plain JSX with no type annotations.
- All layout is done with inline styles using flexbox/grid. No CSS classes except the global reset in `tokens.css`.
- Pages are full-width (`minHeight: '100vh'`, flex column with Nav + scrollable content + Footer). `maxWidth` constraints are applied per-section inside pages, not at the page level.
- Active nav link detection uses `useLocation()` with `startsWith` checks in `Nav.jsx`.
- `useSearchParams` is used for wiki search state (`?q=...`) so search is URL-shareable.
- Wiki article TOC uses `IntersectionObserver` with `rootMargin: '-10% 0px -70% 0px'` to track the active section as the user scrolls.
- Reading progress bar on wiki articles is a `position: fixed; top: 0` lavender bar, 2px tall, `width` driven by scroll percentage.
- Git commits in this environment require `git -c commit.gpgsign=false commit` due to a missing signing server.

## Session Rules — Always Follow These

### Before Every Task
1. Read AGENTS.md fully before doing anything
2. Create a detailed plan with a checklist
3. List every file that will be changed
4. Be 95% confident the approach is correct before executing
5. Do not write any code until the plan is solid

### During Every Task
- Execute one fix at a time
- Check it works before moving to the next item
- Tick off checklist items as completed
- If something is unclear — stop and think, don't guess
- If confidence drops below 95% — stop and re-plan

### Never Do These
- Never write code before planning
- Never change multiple things at once
- Never assume something works — verify it
- Never break existing working features
- Never change colors, fonts, or design tokens
- Never add a max-width wrapper to the page element

### If Something Goes Wrong
- Stop immediately
- Do not try to fix it in place
- Re-read AGENTS.md
- Re-plan the approach
- Execute the corrected plan cleanly
